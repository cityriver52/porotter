# 投稿件数が増えた場合の性能検証

## 結論

今回の改善で、初期表示が読み取るセル数を約65%削減し、スター切替・編集などのID検索は全件取得から該当1行だけの取得へ変更した。

日常的な個人利用では5,000件までは現構成のままで運用しやすい。10,000件を超えたら実環境の応答時間を定期確認し、20,000件前後を目安に古い投稿のアーカイブを検討する。50,000件では初期表示だけで約74万セルを読むため、Googleスプレッドシート1冊を毎回全件検索する設計の限界が見え始める。

## 測定方法

`scripts/benchmark.mjs` は本番データを書き換えず、実際の `Config.gs`、`Repository.gs`、`Api.gs` を疑似Spreadsheet上で実行する。投稿の25%に返信、1%に論理削除を設定し、各条件を5回実行した中央値を記録した。

```powershell
npm.cmd run benchmark
```

時間はローカルPC上のJavaScript処理時間であり、GASウェブアプリの実時間ではない。Google側の混雑、Spreadsheetサービスとの通信、ネットワークによって実時間は変わるため、セル読取数と件数に対する倍率を主な判断材料とする。

## 改善前後

### 初期表示

| 投稿数 | 改善前 | 改善後 | 読取セル削減 | ローカル時間削減 |
|---:|---:|---:|---:|---:|
| 1,000 | 42,558セル / 14.40ms | 14,799セル / 9.24ms | 65.2% | 35.8% |
| 5,000 | 212,558セル / 68.03ms | 73,799セル / 38.34ms | 65.3% | 43.6% |
| 10,000 | 425,058セル / 145.91ms | 147,549セル / 77.91ms | 65.3% | 46.6% |
| 20,000 | 850,058セル / 244.39ms | 295,049セル / 162.36ms | 65.3% | 33.6% |
| 50,000 | 2,125,058セル / 762.49ms | 737,549セル / 387.18ms | 65.3% | 49.2% |

初期表示では、従来は投稿と返信を別シートから複数回読み取っていた。改善後はEntriesを1回読み、タイムライン、Today's discovery、タグ集計で同じスナップショットを再利用する。また、既に正しいシート見出しを毎回書き直す処理を停止した。

### 通常のタイムライン・検索

| 投稿数 | 改善後の通常表示 | 改善後の検索 | 1,000件比の検索時間 |
|---:|---:|---:|---:|
| 1,000 | 14,759セル / 7.57ms | 14,759セル / 7.09ms | 1.0倍 |
| 5,000 | 73,759セル / 26.27ms | 73,759セル / 24.66ms | 3.5倍 |
| 10,000 | 147,509セル / 65.49ms | 147,509セル / 54.03ms | 7.6倍 |
| 20,000 | 295,009セル / 122.11ms | 295,009セル / 119.22ms | 16.8倍 |
| 50,000 | 737,509セル / 295.74ms | 737,509セル / 302.14ms | 42.6倍 |

検索、タグ集計、返信数集計は引き続き全件走査なので、遅延はほぼ投稿数に比例する。ページ表示が20件でも、検索前に全投稿を読む点は変わらない。

### スター切替・編集などのID検索

| 投稿数 | 改善前 | 改善後 |
|---:|---:|---:|
| 10,000 | 130,000セル取得 | 13セル取得 + ID列のTextFinder |
| 50,000 | 650,000セル取得 | 13セル取得 + ID列のTextFinder |

ID列だけをSpreadsheet側のTextFinderで検索し、該当行だけ取得する。ブラウザ側へ移すデータ量は投稿数によらず一定になった。

## 運用上の目安

| 投稿数 | 判断 | 対応 |
|---:|---|---|
| 〜5,000 | 通常運用 | 現構成で継続。体感が悪化した時だけ実測する |
| 5,000〜10,000 | 観察開始 | 初期表示・検索の実時間を月1回程度確認する |
| 10,000〜20,000 | 遅延が見えやすい | 返信やタグを含む全件読取が15万〜30万セル。アーカイブ方針を決める |
| 20,000〜50,000 | 構成変更を推奨 | 年ごとのアーカイブ、集計シート、検索索引のいずれかを導入する |
| 50,000〜 | 現設計の上限域 | Sheets全件走査をやめ、直近データとアーカイブを分離する |

これは失敗する固定上限ではなく、遅延が増える設計上の境界である。Apps Scriptの公式上限は現在1実行6分だが、ユーザーが待てる時間はそれよりはるかに短い。Googleもサービス呼出しを減らし、Spreadsheetの読み書きをまとめることを推奨している。

## 次に効く対策

1. **年単位のアーカイブ**  
   Entriesのうち古い年を別シートまたは別スプレッドシートへ移す。普段のタイムラインは直近1〜2年だけを対象にし、検索時だけアーカイブを選べるようにする。個人利用では最も単純で壊れにくい。

2. **タグ・返信数の集計を別シートへ保持**  
   投稿や返信の作成・削除時に集計値を更新し、初期表示時の全件走査を減らす。書込み処理と集計値の整合性を保つ仕組みが必要になる。

3. **直近ページを末尾から読む**  
   ホーム画面は末尾の必要行だけ取得し、検索・タグ・Today's discoveryは必要になった時に別APIで遅延読込みする。初期表示は件数によらずほぼ一定にできる一方、画面側の変更が大きい。

4. **検索索引を追加**  
   2万件を超えて横断検索を多用する場合、検索語・タグ・投稿IDを持つ索引シートを作る。さらに増える場合はSheetsをデータベース代わりに使う構成自体を見直す。
